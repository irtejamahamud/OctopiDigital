üìñ Git & GitHub Basics

What is Git?

Git is a distributed version control system that tracks changes in files and coordinates work among multiple developers. It allows you to save snapshots of your project at different points in time.

Key Concepts:
  • Repository (Repo): A folder containing your project files and Git history
  • Commit: A snapshot of your project at a specific time
  • Branch: A parallel version of your repository
  • Merge: Combining changes from different branches
  • Remote: A version of your repository hosted on a server

What is GitHub?

GitHub is a cloud-based platform that hosts Git repositories and provides additional collaboration features like issue tracking, pull requests, and project management tools.

Git vs GitHub

Git

  • Version control system
  • Works locally on your machine
  • Command-line tool
  • Tracks file changes
  • Open source and free

GitHub

  • Cloud hosting platform
  • Web-based interface
  • Social coding features
  • Issue tracking
  • Collaboration tools

The Three States of Git

1. Working Directory

Files you're currently editing. These changes are not tracked by Git yet.

2. Staging Area (Index)

Files that are ready to be committed. You add files here with 'git add'.

3. Repository

Committed files that are permanently saved in Git history.

Setup & Configuration

# Configure your identity git config --global user.name "Your Name" git config --global user.email "your.email@example.com" # Check configuration git config --list # Get help git help <command> git <command> --help

Repository Operations

# Initialize a new repository git init # Clone an existing repository git clone <repository-url> git clone <repository-url> <directory-name> # Check repository status git status # View commit history git log git log --oneline git log --graph --oneline --all

Basic Workflow Commands

# Add files to staging area git add <file-name> git add . # Add all files git add *.txt # Add all .txt files git add -A # Add all changes # Commit changes git commit -m "Commit message" git commit -am "Add and commit in one step" # View differences git diff # Working directory vs staging git diff --staged # Staging vs last commit git diff HEAD # Working directory vs last commit

Branch Management

# List branches git branch # Local branches git branch -r # Remote branches git branch -a # All branches # Create and switch branches git branch <branch-name> git checkout <branch-name> git checkout -b <branch-name> # Create and switch # Modern way (Git 2.23+) git switch <branch-name> git switch -c <branch-name> # Create and switch # Merge branches git merge <branch-name> # Delete branches git branch -d <branch-name> # Safe delete git branch -D <branch-name> # Force delete

Remote Repository Commands

# Add remote repository git remote add origin <repository-url> # View remotes git remote -v # Push changes git push origin <branch-name> git push -u origin <branch-name> # Set upstream # Pull changes git pull origin <branch-name> git pull # From current branch's upstream # Fetch changes (without merging) git fetch origin

Undoing Changes

# Unstage files git reset <file-name> git reset # Unstage all # Discard working directory changes git checkout -- <file-name> git restore <file-name> # Modern way # Reset to previous commit git reset --soft HEAD~1 # Keep changes staged git reset --mixed HEAD~1 # Keep changes in working directory git reset --hard HEAD~1 # Discard all changes # Revert a commit (safe way) git revert <commit-hash>

Advanced Commands

# Stash changes temporarily git stash git stash push -m "message" git stash list git stash apply git stash pop git stash drop # Rebase (rewrite commit history) git rebase <branch-name> git rebase -i <commit-hash> # Interactive rebase # Cherry-pick specific commits git cherry-pick <commit-hash> # Tag releases git tag <tag-name> git tag -a <tag-name> -m "message" git push origin <tag-name>

üéØ Real-World Scenarios

Scenario 1: Starting a New Project

Situation:

You're starting a new web development project and want to track it with Git.

Steps:

# 1. Create project directory mkdir my-awesome-project cd my-awesome-project # 2. Initialize Git repository git init # 3. Create initial files echo "# My Awesome Project" > README.md echo "node_modules/" > .gitignore # 4. Add and commit git add . git commit -m "Initial commit: Add README and gitignore" # 5. Connect to GitHub (optional) git remote add origin https://github.com/username/my-awesome-project.git git push -u origin main

Scenario 2: Collaborating on Existing Project

Situation:

You need to contribute to an existing project on GitHub.

Steps:

# 1. Fork the repository on GitHub (via web interface) # 2. Clone your fork git clone https://github.com/your-username/project-name.git cd project-name # 3. Add original repository as upstream git remote add upstream https://github.com/original-owner/project-name.git # 4. Create feature branch git checkout -b feature/new-awesome-feature # 5. Make changes and commit git add . git commit -m "Add new awesome feature" # 6. Push to your fork git push origin feature/new-awesome-feature # 7. Create Pull Request on GitHub

Scenario 3: Fixing a Bug in Production

Situation:

Critical bug found in production. Need to fix quickly without disrupting ongoing development.

Steps:

# 1. Create hotfix branch from main/master git checkout main git pull origin main git checkout -b hotfix/critical-bug-fix # 2. Fix the bug # (edit files) # 3. Test and commit git add . git commit -m "Fix critical authentication bug" # 4. Merge back to main git checkout main git merge hotfix/critical-bug-fix # 5. Deploy to production git push origin main # 6. Merge into develop branch too git checkout develop git merge hotfix/critical-bug-fix git push origin develop # 7. Delete hotfix branch git branch -d hotfix/critical-bug-fix

Scenario 4: Merge Conflicts

Situation:

Two developers modified the same file. Git can't automatically merge.

Steps:

# 1. Attempt to merge (conflict occurs) git merge feature-branch # 2. Check conflicted files git status # 3. Open conflicted file and resolve # Look for conflict markers: # <<<<<<< HEAD # Your changes # ======= # Their changes # >>>>>>> feature-branch # 4. After resolving conflicts git add <resolved-file> git commit -m "Resolve merge conflict in <file>"

Scenario 5: Accidentally Committed Wrong Files

Situation:

You committed sensitive files (passwords, large files) that shouldn't be in the repository.

Steps:

# If not pushed yet: # 1. Remove file from staging git reset HEAD <file> # 2. Remove file from working directory rm <file> # 3. Amend the commit git commit --amend # If already pushed (more complex): # 1. Remove file from repository git rm --cached <file> # 2. Add to .gitignore echo "<file>" >> .gitignore # 3. Commit the removal git commit -m "Remove sensitive file" git push origin main # 4. For complete removal from history (dangerous): git filter-branch --force --index-filter \ 'git rm --cached --ignore-unmatch <file>' \ --prune-empty --tag-name-filter cat -- --all

Scenario 6: Working with Multiple Features

Situation:

You're working on multiple features simultaneously and need to switch between them.

Steps:

# 1. Working on feature A git checkout -b feature/user-authentication # (make changes) git add . git commit -m "WIP: Add login form" # 2. Urgent bug needs fixing git stash push -m "WIP on authentication" git checkout main git checkout -b hotfix/login-bug # 3. Fix bug and commit git add . git commit -m "Fix login validation bug" # 4. Back to feature work git checkout feature/user-authentication git stash pop # 5. Continue feature development # (make more changes) git add . git commit -m "Complete user authentication feature"

Creating a Pull Request

Step-by-Step Process:

  1. Fork the repository (if not a collaborator)
  2. Create a feature branch
  3. Make your changes and commit
  4. Push to your fork/branch
  5. Create PR through GitHub interface
  6. Fill in PR template with:
    • Clear description of changes
    • Link to related issues
    • Screenshots (if UI changes)
    • Testing instructions
  7. Request reviewers
  8. Address feedback and make changes
  9. Merge when approved

Code Review Standards

What to Review:

  • Functionality: Does the code work as intended?
  • Performance: Are there any performance issues?
  • Security: Any security vulnerabilities?
  • Maintainability: Is the code clean and readable?
  • Testing: Are there adequate tests?
  • Documentation: Is it properly documented?

Commit Message Standards

# Conventional Commits Format <type>[optional scope]: <description> [optional body] [optional footer(s)] # Examples: feat: add user authentication system fix: resolve login validation bug docs: update API documentation test: add unit tests for user service refactor: optimize database queries style: fix code formatting issues chore: update dependencies

Security Best Practices

Essential Security Measures:

  • SSH Keys: Use SSH keys instead of passwords for authentication
  • Two-Factor Authentication: Enable 2FA on all accounts
  • Access Tokens: Use personal access tokens for API access
  • Secret Scanning: Never commit passwords, API keys, or certificates
  • Branch Protection: Prevent direct pushes to main/master
  • Signed Commits: Use GPG signing for commit verification

Team Collaboration Guidelines

Daily Workflow:

  3. Regular Commits: Commit frequently with clear messages
  4. Code Review: All changes go through pull requests
  5. Testing: Ensure all tests pass before merging
  6. Documentation: Update docs with significant changes

üîÑ Professional Git Workflow

Complete Development Workflow

1. Project Setup Phase

# Initialize project repository git init git remote add origin https://github.com/company/project.git # Set up branch protection and team access # (Done through GitHub web interface) # Create development branch git checkout -b develop git push -u origin develop

2. Feature Development Phase

# Start new feature git checkout develop git pull origin develop git checkout -b feature/user-dashboard # Development cycle git add . git commit -m "feat: implement user dashboard layout" git push origin feature/user-dashboard # Create pull request through GitHub interface # Request code review from team members

3. Code Review Phase

# Address reviewer feedback git add . git commit -m "fix: address code review comments" git push origin feature/user-dashboard # Sync with latest develop changes git checkout develop git pull origin develop git checkout feature/user-dashboard git rebase develop git push --force-with-lease origin feature/user-dashboard

4. Integration Phase

# After PR approval, merge to develop git checkout develop git pull origin develop git merge --no-ff feature/user-dashboard git push origin develop # Clean up feature branch git branch -d feature/user-dashboard git push origin --delete feature/user-dashboard

5. Release Phase

# Create release branch git checkout develop git pull origin develop git checkout -b release/v1.2.0 # Final testing and bug fixes git add . git commit -m "fix: resolve release bugs" # Merge to main and tag git checkout main git merge --no-ff release/v1.2.0 git tag -a v1.2.0 -m "Release version 1.2.0" git push origin main git push origin v1.2.0 # Merge back to develop git checkout develop git merge --no-ff release/v1.2.0 git push origin develop

Emergency Hotfix Workflow

Critical Production Issue:

# Create hotfix from main git checkout main git pull origin main git checkout -b hotfix/critical-security-fix # Fix the issue git add . git commit -m "hotfix: patch security vulnerability" # Test thoroughly npm test npm run e2e-tests # Deploy hotfix git checkout main git merge --no-ff hotfix/critical-security-fix git tag -a v1.2.1 -m "Hotfix version 1.2.1" git push origin main git push origin v1.2.1 # Merge to develop git checkout develop git merge --no-ff hotfix/critical-security-fix git push origin develop # Clean up git branch -d hotfix/critical-security-fix
